## Palette's Journal\n\n## 2025-02-12 - Replacing div buttons with semantic button elements for toggles\n**Learning:** Found a common pattern where a mobile menu toggle (`<MenuIcon>`) was implemented as a `styled.div` instead of a `styled.button`. While `cursor: pointer` handles mouse users, this approach completely breaks keyboard accessibility since `div`s are not focusable by default and screen readers cannot identify them as interactive controls. \n**Action:** When creating toggle icons (like hamburger menus), always use semantic `styled.button` components with appropriate ARIA attributes (`aria-label`, `aria-expanded`, `aria-controls`) and apply CSS resets (`background: none; border: none; padding: 0; outline: none;`) to achieve the desired visual style without sacrificing accessibility.

## 2025-04-28 - Accessible and UX-friendly custom modals in React
**Learning:** Found that custom modals lacking essential accessibility and UX features are a recurring pattern. Implementing a `styled.div` for a modal overlay without considering user interactions means breaking key expected behaviors. Users expect to close modals using the 'Escape' key and by clicking outside the modal content on the overlay. Moreover, screen readers need to understand the structural context of the modal through `role="dialog"`, `aria-modal="true"`, and appropriate `aria-label`s. Interactive elements inside the modal must also have visible focus states (`:focus-visible`) for proper keyboard navigation.
**Action:** When implementing custom modals, always ensure they include critical UX and accessibility features: Escape key listener, overlay click-to-close behavior (`e.target === e.currentTarget`), appropriate ARIA attributes (`role="dialog"`, `aria-modal="true"`, `aria-label`), and `:focus-visible` styles for close buttons.
